# ADR-0002：百炼向量模型与混合检索

- 状态：已接受
- 日期：2026-07-22
- 关联决策：[ADR-0001](ADR-0001-foundation-stack.md)

## 背景与问题

第二阶段需要将 TXT（纯文本）和 Markdown（轻量级标记语言）资料转换为可检索知识片段。系统既要理解“清爽、酸甜、适合新手”等语义描述，也要稳定命中酒名、材料名和专业术语，因此单独依赖关键词或向量检索都不充分。

## 目标与约束

- 使用用户现有的阿里云百炼北京地域业务空间与 OpenAI-compatible API（OpenAI 兼容接口）。
- 密钥、业务空间地址和模型名从环境变量读取，不提交真实值。
- 当前向量模型固定为 `text-embedding-v4`，输出维度固定为 1024。
- 文档处理失败可观测、可重试，未完成片段不能进入检索结果。
- 自动化测试不得调用真实云端模型，避免费用、限流和不稳定网络影响测试结果。

## 备选方案

### Java 模型接入层

1. LangChain4j（Java 大模型应用框架）稳定核心模块：适合显式编排模型与工具，供应商适配边界清楚，但需要自行完成 Spring（Java 应用框架）装配。
2. Spring AI（Spring 人工智能框架）：与 Spring 生态集成紧密，但本项目更希望展示 Agent（智能体）编排本身，且避免让自动配置隐藏调用边界。
3. 直接使用 HTTP Client（HTTP 客户端）：依赖最少，但请求响应映射、重试和后续模型切换样板代码较多。

### 检索方案

1. 仅向量检索：模糊语义召回较好，但精确术语可能被相似语义挤出。
2. 仅 PostgreSQL（关系型数据库）关键词检索：精确词可解释，但中文连续文本分词和模糊需求能力有限。
3. 关键词与向量两路召回，再使用 RRF（倒数排名融合）：不要求两路原始分数处于同一量纲，便于后续独立替换任一路实现。

## 最终决策

- 使用 LangChain4j 1.18.0 的稳定核心模块和 OpenAI（开放人工智能）模型适配模块，不引入其测试版 Spring Boot Starter（Spring Boot 启动依赖）。
- 通过 `TextEmbeddingPort`（文本向量端口）隔离 LangChain4j 与百炼，应用层测试使用假实现。
- 使用 `text-embedding-v4` 的 1024 维向量；数据库字段固定为 `vector(1024)` 并创建 HNSW（分层可导航小世界图）余弦索引。
- 关键词召回组合精确子串、`pg_trgm`（PostgreSQL 三元组相似度扩展）和内置全文相关度。
- 两路候选使用常数 60 的 RRF 融合，同时保留各自排名和原始分数，用于解释与离线评估。
- 文档处理采用 `UPLOADED -> PARSING -> CHUNKING -> EMBEDDING -> READY` 状态机，失败进入 `FAILED` 并记录安全错误码。

## 负面影响

- 1024 维一旦写入表类型和索引，更换维度需要新迁移、重算全部片段向量并重建索引。
- 当前中文关键词召回是低成本基线，不等同于专业中文搜索引擎的分词质量。
- 进程内有界线程池适合单实例演示；多实例部署时无法提供队列级统一调度。
- RRF 常数和两路候选数尚未通过评估集调优，当前属于待验证参数。

## 风险与缓解措施

- 模型调用设置超时、有限重试、批量上限和调用指标；日志不记录密钥、原文或模型原始响应。
- 检索两路独立执行，任一路失败时降级返回另一路结果，只有两路都失败才返回服务不可用。
- 数据库只允许 `READY` 文档参与检索；片段替换和状态更新处于同一事务。
- 状态更新包含期望旧状态，重复任务只有一个能获得处理权。
- 启动时将超时处理中任务标记失败，并重新调度遗留的 `UPLOADED` 任务，覆盖常见宕机窗口。
- 管理接口使用 HTTP Basic（HTTP 基本认证）；公开部署必须增加 HTTPS（安全超文本传输协议）并升级鉴权方案。

## 验证方式

- 单元测试验证分块边界、片段最大长度和 RRF 融合排序，不调用真实模型。
- Testcontainers（容器化集成测试工具）验证 `vector(1024)`、HNSW 索引和全部 Flyway（数据库迁移工具）脚本。
- 手工上传一份 UTF-8 文档，轮询至 `READY`，再检查关键词、语义描述和混合命中的排名证据。
- 使用错误编码文件、重复文件、缺失密钥和非失败状态重试验证异常协议。

## 适用边界与未来演进

当前方案面向单实例、低并发和中小规模知识库。出现以下实测信号后再升级：

- 中文关键词召回评估长期不达标：引入 Elasticsearch/OpenSearch（搜索引擎）并保持应用层端口不变。
- 向量检索拖慢事务业务或 P95（95% 请求耗时分位数）超标：迁移到独立向量服务。
- 多实例任务重复、任务积压或恢复时间不可接受：引入持久化任务表轮询或消息队列。
- 更换模型或维度：以新列或新表重建索引，完成双读评估后再切换，禁止混用不同模型向量。
