# 第一阶段数据库结构

## 数据边界

结构化配方数据与 RAG 知识数据分开保存：

- `ingredient`：酒材标准名称、类别、酒精体积分数和过敏提示。
- `recipe`：配方说明、制作步骤、难度和来源。
- `recipe_ingredient`：配方与酒材关系、用量、单位和处理说明。
- `knowledge_document`：上传文档的元数据、版本和处理状态，不直接保存文件二进制内容。
- `knowledge_chunk`：文档分块、检索元数据、全文检索字段和向量。

## 单位与酒精度

配方保留原始单位，液体参与总体积和酒精度计算前必须换算为毫升。酒精体积分数只能来自经过确认的酒材数据，不能由大模型猜测。

## 知识文档状态

文档处理采用明确状态：

```text
UPLOADED -> PARSING -> CHUNKING -> EMBEDDING -> READY
                    \-> FAILED
READY -> DISABLED
```

状态用于支持页面查询进度、失败重试和防止未完成文档进入检索链路。状态转换使用期望旧状态作为更新条件，重复任务只有一个能获得处理权；片段替换与 `READY` 更新处于同一事务。

## 当前限制

- V3 迁移已将向量固定为 `vector(1024)` 并创建 HNSW（分层可导航小世界图）余弦索引，对应百炼 `text-embedding-v4` 当前配置。模型或维度变更必须重算全部向量。
- 当前全文字段使用 PostgreSQL `simple` 配置，只作为混合检索基础，不代表已经解决中文分词质量问题。
- 本地 Docker 数据库用户可以创建扩展；生产环境若使用受限账号，需要由数据库管理员预先安装 `vector` 和 `pg_trgm` 扩展。
- 用户、库存、偏好、推荐轨迹和反馈表将在对应用例设计完成后通过后续 Flyway 迁移加入。
